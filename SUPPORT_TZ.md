# Техническое Задание (ТЗ) на поддержку проекта "Joyseekers Community Bot"

## 1. Общее описание проекта
Проект представляет собой Telegram-бот для закрытого международного комьюнити. Основная цель бота — позволить участникам делиться ресурсами и услугами, находить нужные контакты и специалистов, а также осуществлять обмен внутри комьюнити на основе балльной системы.

Бот (имя: `JOYSEEKERS`) предоставляет сложную систему онбординга (анкетирования) из 9 секций, систему предложений и запросов (Лоты) и базу открытых ресурсов (Карты, специалисты).

## 2. Стек технологий
* **Язык**: Python 3.8+
* **Фреймворк**: `aiogram` 3.4.1 (асинхронный Telegram Bot API)
* **База данных**: SQLite
* **Драйвер БД**: `aiosqlite` (асинхронная работа с БД)
* **Конфигурация**: `python-dotenv` (переменные окружения)
* **Инфраструктура**: Проект деплоится на Railway (или аналогичных PaaS).

## 3. Структура проекта
Исходный код бота находится в директории `bot/`, запуск осуществляется через `run.py`.

* `bot/main.py` — Точка входа, инициализация бота, диспетчера (Dispatcher), базы данных и подключение роутеров.
* `bot/config.py` — Настройки конфигурации (токен, парсинг списка администраторов из `ADMIN_IDS`).
* `bot/database.py` — Класс `Database` с асинхронными методами для работы с SQLite.
* `bot/keyboards.py` — Хелперы для генерации всех клавиатур (Reply и Inline), включая кастомные кнопки "Back".
* `bot/form_data.py` — Статические данные: списки городов, категории ресурсов, типы результатов, навыки (ALL_SKILLS) и т.д.
* `bot/handlers/` — Роутеры (обработчики команд и сообщений), разбитые по логике:
  * `registration.py` — Флоу регистрации и анкетирования (FSM).
  * `menu.py` — Главное меню, просмотр профиля.
  * `friends.py` — Раздел "Friends -> Locations" (участники по городам).
  * `resources.py` — Раздел ресурсов по категориям.
  * `lots.py` — Раздел лотов (Создать предложение / запрос, просмотр).
  * `deals.py` — Логика сделок.
  * `open_resources.py` — Раздел "MAPS" (карты городов).
  * `admin.py` — Админ-панель (управление пользователями, ресурсами, баллами).

## 4. Архитектура и основные паттерны

### 4.1. Управление состоянием (State Management)
* Бот активно использует **FSM (Finite State Machine)** от aiogram (`MemoryStorage`) для процессов регистрации, создания лотов и добавления ресурсов.
* Временное состояние, не требующее FSM, может храниться в in-memory Python-словарях.
* `await state.clear()` обязательно вызывается в хендлере `/start` для сброса состояния.

### 4.2. База Данных (SQLite)
* Бот использует SQLite. Объект базы данных пробрасывается во все хендлеры через middleware (`db_middleware` в `main.py`).
* Таблицы: `users`, `resources`, `lots`, `open_resources`, `invite_tokens`, `deals`, `user_answers`.
* **ВАЖНОЕ ОГРАНИЧЕНИЕ ПО ДЕПЛОЮ:** При деплое на Railway или аналогичные сервисы, SQLite файл создается на эфемерной файловой системе. Без настройки персистентного хранилища (Volume) база данных будет стираться при каждом редеплое. Разработчик поддержки должен следить за сохранностью БД.

### 4.3. Навигация и UI
* Для перехода между секциями используются `edit_text` или `edit_reply_markup`, чтобы не засорять историю чата.
* Переход от `ReplyKeyboardMarkup` к `InlineKeyboardMarkup` обязательно сопровождается отправкой сообщения с `ReplyKeyboardRemove()`.
* Во всех Callback-хендлерах должна использоваться конструкция `try...except...finally` с обязательным вызовом `await callback.answer()` в `finally`, чтобы избежать вечной загрузки интерфейса Telegram.
* Длинные данные в Inline-кнопках (например, названия навыков) хэшируются (MD5), так как Telegram ограничивает длину `callback_data` 64 байтами.
* При нажатии на инлайн-кнопки (особенно в регистрации) они часто удаляются из сообщения, чтобы предотвратить двойные клики и нарушение консистентности FSM.

### 4.4. Специфика флоу регистрации
* Вход возможен только по инвайт-коду ("JOY").
* Данные анкеты формируются из статики в `form_data.py`. Документация по текстовым формулировкам лежит в `form.md`.
* Некоторые шаги (например, выбор вместимости в секции "Real Estate") настроены на авто-переход (auto-advance) к следующему вопросу сразу после выбора, без кнопки "Done".
* Раздел навыков ("Skills and Knowledge") требует выбора категории, затем конкретных навыков, и возврата "Назад" (Back) к списку категорий для выбора других навыков.
* Для предотвращения состояния "гонки" (race conditions) при двойных кликах используются флаги процессинга (processing flags) в данных FSM.

## 5. Известные проблемы и предупреждения
* **TelegramConflictError:** Запуск более чем одного инстанса бота с одним и тем же токеном (например, на проде и локально одновременно) приведет к конфликту и непредсказуемому поведению.
* **Fallbacks:** В `deals.py` реализован fallback-хендлер для текстового сообщения "Back" на случай, если у пользователя слетел FSM-стейт, чтобы перенаправить его в Главное меню. Аналогичные обработки `strip()` используются в кнопках "Back" везде по коду.
* Хардкод: Статические данные (города, категории) зашиты в код (`form_data.py`), а не вынесены в БД. При добавлении нового города или категории требуется обновление кода и рестарт бота. Отображаемые имена (с эмодзи, например, 🖤 Cars) отделены от внутренних ключей в БД.

## 6. Задачи по поддержке
Разработчику, берущему проект на поддержку, предстоит:

1. **Мониторинг логов и стейтов:** Отслеживание ошибок FSM, потери стейтов при таймаутах сессий.
2. **Обновление контента:** Регулярное обновление списков городов, категорий навыков, типов результатов в `bot/form_data.py`. Формулировки должны быть короткими, без скобок (например, "и т.д." вместо скобок).
3. **Модерация и Админ-панель:** Поддержка функционала модерации лотов и изменения баллов пользователей через `/admin`. (ID админов парсятся из ENV).
4. **Инфраструктура:** Контроль сохранности файла SQLite базы данных при деплоях, создание бекапов.
5. **Развитие функционала:** Добавление новых шагов в анкету при необходимости, настройка интеграции с внешними системами (в перспективе).
